================================================================================
PROJECT HISTORY & ARCHITECTURAL LOG
================================================================================

This file contains the key development history, user instructions, and technical responses detailing the architectural decisions of this codebase. Read this file to instantly understand the context of the CodeBase Q&A project.

--------------------------------------------------------------------------------
1. PROJECT INITIALIZATION & GOALS
--------------------------------------------------------------------------------
[Prompt]
Initialize a codebase Q&A application based on the goals defined in final_goal.txt:
1. Answer code location queries with exact file references and line numbers.
2. Index local folders and remote public GitHub repositories.
3. Automatically draft README sections from the scanned codebase.

[Response Summary]
Designed and implemented a full-stack local web application:
- Backend: Node.js + Express server to handle local scanning, git cloning, and Gemini API query construction.
- Frontend: Single-page application using vanilla HTML, CSS, and Javascript.
- Context Strategy: Fits repository content into Gemini's prompt context, accompanied by the directory file structure.

--------------------------------------------------------------------------------
2. REMOTE REPOSITORY COMPATIBILITY
--------------------------------------------------------------------------------
[Prompt]
Clarify the repository indexing workflow for remote repositories: Will cloning a public GitHub URL result in the identical analysis and output formats as scanning a local directory?

[Response Summary]
Confirmed that remote GitHub links undergo a local `git clone` into a temporary directory (`temp_repos/`) on the server. Once cloned locally, the files are recursively scanned and queried using the exact same pipeline as local directories.

--------------------------------------------------------------------------------
3. API KEY SECURITY AND BACKEND STORAGE
--------------------------------------------------------------------------------
[Prompt]
Secure the Gemini API key by storing it on the backend. The key must not be exposed to the client-side UI, nor must it be committed to version control. Set up a secure configuration environment.

[Response Summary]
Re-architected the key management system:
- Removed API key inputs from the frontend.
- Created a `.env` file at the project root for local backend storage.
- Added `.env` and the `temp_repos/` directories to `.gitignore` to prevent any exposure on GitHub.
- Exposed `/api/status` to let the frontend verify if the API key is configured on the backend.

--------------------------------------------------------------------------------
4. UI PALETTE AND AESTHETIC COMPLIANCE
--------------------------------------------------------------------------------
[Prompt]
Apply a professional, clean, user-friendly design palette (slate, teal, and cyan) avoiding any purple colors in the interface. Enable local configuration of the API key in the `.env` template so it can be added later.

[Response Summary]
Developed a modern slate-dark user interface using CSS glassmorphism, rounded panels, and a strict slate/teal/cyan palette (completely removing all purple tones). Created the complete app files structure including Prism.js integration for syntax highlighting in the Code Viewer.

--------------------------------------------------------------------------------
5. API LOAD AND TOKEN OPTIMIZATION
--------------------------------------------------------------------------------
[Prompt]
Optimize the application to reduce Gemini API load and manage token consumption. Implement API-request caching and limit context sizes.

[Response Summary]
Implemented four major backend API load optimizations:
- Context Size Cap: Reduced the prompt code context limit from 1.2MB to 250KB.
- Retrieval File Limit: Capped the maximum files sent in a single query prompt to the top 6 ranked files.
- In-Memory Query Cache: Added `queryCache` Map to save and return identical query responses instantly.
- README Draft Cache: Added `readmeCache` Map to optimize duplicate README generation runs.
- All caches are cleared automatically when a new repository is indexed.

--------------------------------------------------------------------------------
6. CACHE RESET ON REFRESH & IN-VIEWER FILE EXPLORER
--------------------------------------------------------------------------------
[Prompt]
Clear the indexed codebase files and statistics from the UI on browser refresh. In the Code Viewer tab, implement a file selection dropdown so users can open and switch between any indexed file directly within the viewer.

[Response Summary]
- In-Viewer File Selection: Replaced the static header in the Code Viewer with a styled `<select>` dropdown (`.viewer-select`). Populated the dropdown dynamically with all indexed files upon successful scanning, enabling quick file switching inside the viewer tab.

--------------------------------------------------------------------------------
7. WORKSPACE LAYOUT & DIRECTORY TREE DESIGN SYSTEM
--------------------------------------------------------------------------------
[Prompt]
Enhance the visual design and usability of the interface:
1. Split Pane Workspace: Combine Chat and Code Viewer side-by-side inside a single Q&A Workspace tab. When file reference badges are clicked, the code opens on the right side instantly without losing chat context.
2. Nested Folder Tree: Replace the flat list of files with a collapsible/expandable nested folder directory structure tree.
3. Marked.js Markdown rendering: Add Marked.js parsing to format headers, lists, code snippets, and tables natively.
4. Glowing Line Range Highlights: Support highlighting specific lines in the viewer and scrolling them smoothly into view when clicked from code reference tags.

[Response Summary]
- Workspace split layout: Re-structured `index.html` and `index.css` to group Chat panel (left) and Code Viewer (right) side-by-side, creating a productive IDE-like split screen.
- Directory Tree View: Implemented recursive tree parsing (`buildFileTree`, `renderTree`) in `app.js` with toggleable expand states and styled directory icons. Supports flat fallback listing during search filters.
- Marked.js integration: Loaded Marked CDN and integrated `marked.parse()` to render professional markdown outputs in chat.
- Glowing lines scroll: Added parameter scanning for lines (`#L45-L60`) in badge click handlers. Programmed `loadFileInViewer` to apply a glowing cyan line highlight class (`.highlighted-line-num`) to the active ranges and scroll the pre tag smoothly to focus.

--------------------------------------------------------------------------------
8. DRAGGABLE SPLIT PANE RESIZER
--------------------------------------------------------------------------------
[Prompt]
Make the split pane resizable so users can adjust the width of the Chat and Code Viewer panels.

[Response Summary]
- HTML/CSS changes: Placed a `.resizer` drag-handle div between the chat-pane and code-pane elements, and updated the CSS layout from display-grid to display-flex to support fluid width adjustments.
- JS drag handle: Programmed `initResizer` in `app.js` with mousedown/mousemove/mouseup listeners. Dynamically recalculates the left panel's width on-drag with smooth boundaries (min 280px, max 75%).

--------------------------------------------------------------------------------
9. DYNAMIC UI COLOR THEME SELECTOR
--------------------------------------------------------------------------------
[Prompt]
Enable trying multiple UI color schemes to find the most visually pleasing layout.

[Response Summary]
- CSS Variables mapping: Configured alternative color variable values under `body[data-theme]` attributes in `index.css` for: GitHub Dark (Steel Blue), Nord Frost (Icy Arctic), One Dark (Warm Charcoal), and Forest Hacker (Sage Emerald).
- Theme dropdown selector: Inserted a styled `<select>` element (`#theme-select`) inside the sidebar in `index.html`.
- LocalStorage persistence: Added `changeTheme` and `restoreSavedTheme` in `app.js` to change the theme dynamically on select change, save the selection in `localStorage`, and auto-restore the user's preference on reload.
