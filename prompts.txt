================================================================================
PROJECT DEVELOPMENT LOG — SourceInsight (CodeBase Q&A)
================================================================================

This document chronicles the full architectural evolution of the SourceInsight
codebase analyzer. Each entry records the development prompt, the technical
decisions made, and the implementation outcomes across 17 iterative stages.

================================================================================


--------------------------------------------------------------------------------
1. PROJECT INITIALIZATION & GOALS
--------------------------------------------------------------------------------
[Prompt]
Design and implement a codebase Q&A application capable of answering developer
questions about any repository with exact file references and line numbers.
The system must support indexing both local directories and remote public GitHub
repositories. Additionally, provide functionality to auto-generate professional
README sections from the scanned codebase content.

[Response Summary]
Designed and implemented a full-stack local web application:
- Backend: Node.js + Express server handling recursive file scanning, temporary
  git cloning, and Gemini API prompt construction.
- Frontend: Single-page application using vanilla HTML, CSS, and JavaScript.
- Context Strategy: Fits repository content into Gemini's prompt context window,
  accompanied by the full directory file structure for spatial awareness.


--------------------------------------------------------------------------------
2. REMOTE REPOSITORY COMPATIBILITY
--------------------------------------------------------------------------------
[Prompt]
Ensure the repository indexing pipeline produces identical analysis and output
formats regardless of whether the input source is a local directory path or a
remote public GitHub URL.

[Response Summary]
Confirmed uniform pipeline behavior: Remote GitHub URLs undergo a local
`git clone` into a temporary directory (`temp_repos/`) on the server. Once
cloned, files are recursively scanned and queried using the exact same
processing pipeline as local directories, ensuring output parity.


--------------------------------------------------------------------------------
3. API KEY SECURITY & BACKEND STORAGE
--------------------------------------------------------------------------------
[Prompt]
Secure the Gemini API key by storing it exclusively on the backend. The key must
never be exposed to the client-side UI or committed to version control. Set up a
secure environment configuration system.

[Response Summary]
Re-architected the key management layer:
- Removed all API key input fields from the frontend interface.
- Created a `.env` file at the project root for backend-only storage.
- Added `.env` and `temp_repos/` to `.gitignore` to prevent repository exposure.
- Exposed a `/api/status` endpoint to allow the frontend to verify API key
  configuration status without transmitting the key itself.


--------------------------------------------------------------------------------
4. UI PALETTE & AESTHETIC COMPLIANCE
--------------------------------------------------------------------------------
[Prompt]
Apply a professional, clean, developer-friendly design palette using slate, teal,
and cyan tones. Eliminate all purple-toned colors from the interface. Enable
local configuration of the API key through a `.env` template.

[Response Summary]
Developed a modern slate-dark user interface featuring CSS glassmorphism, rounded
panels, and a strict slate/teal/cyan palette. Integrated Prism.js for syntax
highlighting in the Code Viewer. Created the complete frontend asset structure.


--------------------------------------------------------------------------------
5. API LOAD & TOKEN OPTIMIZATION
--------------------------------------------------------------------------------
[Prompt]
Optimize the application to reduce Gemini API token consumption and minimize
redundant API calls. Implement request caching and enforce context size limits.

[Response Summary]
Implemented four backend API load optimizations:
- Context Size Cap: Reduced prompt code context limit from 1.2MB to 250KB.
- Retrieval File Limit: Capped maximum files per query prompt to the top 6
  ranked files by relevance.
- In-Memory Query Cache: Added a `queryCache` Map to store and return identical
  query responses instantly.
- README Draft Cache: Added a `readmeCache` Map to optimize duplicate README
  generation runs.
- All caches are cleared automatically when a new repository is indexed.


--------------------------------------------------------------------------------
6. CACHE RESET ON REFRESH & IN-VIEWER FILE EXPLORER
--------------------------------------------------------------------------------
[Prompt]
Clear the indexed codebase files and statistics from the UI on browser refresh.
In the Code Viewer tab, implement a file selection dropdown so users can browse
and switch between any indexed file directly within the viewer.

[Response Summary]
- In-Viewer File Selection: Replaced the static Code Viewer header with a styled
  `<select>` dropdown (`.viewer-select`), populated dynamically with all indexed
  files upon successful scanning, enabling instant file switching.


--------------------------------------------------------------------------------
7. WORKSPACE LAYOUT & DIRECTORY TREE DESIGN SYSTEM
--------------------------------------------------------------------------------
[Prompt]
Enhance the visual design and usability of the interface with four improvements:
1. Split Pane Workspace: Combine Chat and Code Viewer side-by-side inside a
   single Q&A Workspace tab. Clicking file reference badges should open code on
   the right side without losing chat context.
2. Nested Folder Tree: Replace the flat file list with a collapsible/expandable
   nested directory structure tree.
3. Marked.js Markdown Rendering: Add Marked.js parsing for headers, lists, code
   snippets, and tables in chat responses.
4. Glowing Line Range Highlights: Support highlighting specific line ranges in
   the viewer and scrolling them smoothly into view from code reference tags.

[Response Summary]
- Workspace Split Layout: Restructured `index.html` and `index.css` to group
  Chat (left) and Code Viewer (right) side-by-side in a productive IDE-like
  split-screen arrangement.
- Directory Tree View: Implemented recursive tree parsing (`buildFileTree`,
  `renderTree`) in `app.js` with toggleable expand/collapse states and styled
  directory icons. Supports flat fallback listing during search filtering.
- Marked.js Integration: Loaded Marked CDN and integrated `marked.parse()` to
  render professional markdown outputs in the chat panel.
- Glowing Line Scroll: Added parameter scanning for line ranges (`#L45-L60`) in
  badge click handlers. Programmed `loadFileInViewer` to apply a glowing cyan
  highlight class to active ranges and smooth-scroll the pre tag to focus.

12. PYTHON, STREAMLIT, & LANGCHAIN MIGRATION
--------------------------------------------------------------------------------
[Prompt]
Rewrite the entire codebase analyzer from a Node.js/Express full-stack
application into a Python Streamlit application integrated with LangChain,
Google Gemini Embeddings, and an in-memory FAISS vector store for semantic
retrieval-augmented generation (RAG).

[Response Summary]
- Shifted to Python Runtime: Replaced the Express.js backend and HTML/CSS/JS
  frontend with a unified Streamlit web application (`app.py`) and a standard
  Python dependencies manifest (`requirements.txt`).
- Semantic Indexing (LangChain + FAISS): Integrated
  `RecursiveCharacterTextSplitter` and `GoogleGenerativeAIEmbeddings` with an
  in-memory `FAISS` Vector Store to chunk, embed, and search codebases.
- Grounded Q&A Chat: Replaced keyword-based indexing with LangChain
  `similarity_search`, pulling the top 5 relevant code chunks into Gemini API
  context for grounded, citation-backed answering.
- Cleaned Node.js Assets: Deleted all Node.js source files (`server.js`,
  `package.json`, `package-lock.json`, `public/` directory, `node_modules/`,
  and `test_server.js`).

13. EMBEDDING MODEL FIX & STREAMLIT API COMPATIBILITY
--------------------------------------------------------------------------------
[Prompt]
Resolve multiple runtime errors after migration: the embedding model
`text-embedding-004` returns a 404 NOT_FOUND error, the LLM call
`ChatGoogleGenerativeAI.predict()` raises an AttributeError, the
`st.text_area()` call fails with an unsupported `rows` keyword argument,
and text content disappears in dark mode due to theme color conflicts.

[Response Summary]
- Embedding Model: Replaced `models/text-embedding-004` with
  `models/gemini-embedding-001` after programmatically querying the Gemini API
  to list all available embedding models.
- LLM Invocation: Replaced the deprecated `.predict()` method with the current
  LangChain `.invoke()` API, extracting the response from `.content`.
- Streamlit API Fix: Replaced the unsupported `rows=3` parameter in
  `st.text_area()` with the correct `height=100` parameter.
- Dark Mode Text Fix: Added a CSS rule forcing `color: var(--text-main)` on all
  `[data-testid="stMarkdownContainer"]` elements to prevent theme-inherited
  text color clashes.


--------------------------------------------------------------------------------
14. FREE-TIER RATE LIMIT HANDLING & BATCHED EMBEDDING
--------------------------------------------------------------------------------
[Prompt]
Address Gemini API `RESOURCE_EXHAUSTED` (429) errors caused by the free-tier
rate limit of 100 embedding requests per minute. Implement batched embedding
with inter-batch delays and a visual progress indicator.

[Response Summary]
- Batched Embedding: Restructured the FAISS indexing loop to process documents
  in batches of 20, calling `FAISS.from_documents()` for the first batch and
  `vector_store.add_documents()` for subsequent batches.
- Rate-Limit Delay: Added `time.sleep(1.5)` between batches to throttle
  requests below the 100 requests/minute free-tier ceiling.
- Visual Progress Bar: Integrated `st.progress()` with dynamic text updates
  showing `Embedded X/Y chunks...` so users can monitor indexing status in
  real time.

Replace the sidebar width slider with a mouse-draggable resizer bar between the

16. SIDEBAR FILE TREE EXPLORER
--------------------------------------------------------------------------------
[Prompt]
Replace the sidebar file selection dropdown with an interactive, collapsible
directory tree structure so users can visually navigate the codebase hierarchy
and understand the project layout at a glance.

[Response Summary]
- Tree Builder (`build_tree_dict`): Parses the flat list of `/`-separated file
  paths into a nested dictionary structure of `{files: [], dirs: {}}` nodes.
- Recursive Renderer (`render_tree`): Renders the tree using `st.expander()`
  for directories (with folder emoji icons) and `st.button()` for files (with
  document emoji icons). The currently active file is visually marked with a
  pointer indicator.
- Auto-Expansion Logic: Directory branches containing the currently active file
  automatically expand (`expanded=True`), providing immediate path context.
- CSS Styling: Added sidebar-scoped CSS overrides for secondary buttons to
  render as flat, transparent, monospace-font file tree nodes with hover
  highlights.

17. ROBUST RATE-LIMIT FIX: EXPONENTIAL BACKOFF & TF-IDF FALLBACK
--------------------------------------------------------------------------------
[Prompt]
Implement a permanent, repo-size-independent fix for Gemini API 429
RESOURCE_EXHAUSTED errors during embedding. The solution must handle both
per-minute rate limits and daily quota exhaustion gracefully, ensuring indexing
never fails regardless of repository size or API quota state.

[Response Summary]
- Reduced Batch Size: Changed batch size from 20 to 5 documents per API call
  with 4-second inter-batch delays, yielding ~75 requests/minute (safely under
  the 100/minute free-tier ceiling).
- Exponential Backoff Retry: Wrapped each batch in a retry loop (5 attempts)
  with exponential backoff (10s, 20s, 40s, 80s, 90s). Parses the server's
  `retry in Xs` hint from the error response and uses the longer delay.
- TF-IDF Keyword Fallback: When all retries are exhausted (daily quota fully
  depleted), automatically falls back to `TFIDFRetriever` from scikit-learn,
  building a local keyword-based search index with zero API calls.
- Dual Retriever Query Path: Updated the chat query handler to detect which
  retriever is available (`vector_store` for semantic, `tfidf_retriever` for
  keyword) and route queries accordingly.
- Session State Tracking: Added `retriever_type` flag to session state so the
  UI displays the current search mode (Semantic FAISS vs. Keyword TF-IDF).
- Updated Dependencies: Added `scikit-learn` and `langchain-text-splitters` to
  `requirements.txt` for deployment compatibility.
